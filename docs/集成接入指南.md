# 本地导出（SwiftExport）客户端 · 集成接入指南

> 面向「需要把大数据量导出能力接入自己页面」的业务系统（前端 + 后端）。
> 读完本文你能：① 在网页里一行代码唤起客户端导出；② 让你的服务端按约定吐 Arrow 流，
> 客户端即可零改动地生成任意结构的多层合并 Excel。

---

## 1. 它解决什么问题

导出几十万订单 / 数百万行的 Excel 时，浏览器端生成会卡死，服务端 `POI`/`EasyExcel`
一次性生成又容易 OOM、超时、占满带宽。本客户端把「生成」这一步搬到**用户本机的桌面客户端**：

```
 ┌─────────────┐  ①swiftexport:// 深链   ┌──────────────────┐  ②GET Arrow 流   ┌──────────────┐
 │  业务前端    │ ───────────────────────▶ │  本地导出 客户端   │ ───────────────▶ │  业务后端     │
 │ (任意网页)   │      (浏览器唤起)        │ (Win / macOS)    │ ◀─────────────── │ Arrow 流接口  │
 └─────────────┘                          └──────────────────┘   ③流式回传      └──────────────┘
                                                   │
                                                   ▼ ④本地流式生成 .xlsx（多层合并 + 多文件分块）
                                            用户下载目录
```

- **真流式**：响应体边到边解码、边写盘，**峰值内存与数据总量无关**（靠「每 N 个订单一个文件」分块控内存）。
- **零硬编码**：导出长什么样（标题、列、合并、层级）全部由 **Arrow schema 自描述**，
  新增一种导出**不需要改客户端、不需要发版**。

适用：订单/账单/对账单/报表等「主从多层、需要合并单元格」的大批量导出。

## 2. 前置条件

| 项 | 说明 |
|----|------|
| 客户端已安装 | 用户机器需安装「本地导出」客户端（Windows `.exe` / macOS `.dmg`，见仓库 [Releases](../../releases)）。安装后自动注册 `swiftexport://` 协议。 |
| 已设下载目录 | 用户首次使用需在客户端「设置」里选下载目录；未设置时点击导出会自动弹出设置面板。 |
| 后端可吐 Arrow 流 | 你的服务端需提供一个返回 **Arrow IPC Stream** 的 HTTP 接口（见 [第 4 节](#4-后端接入arrow-流式导出接口契约)）。 |

## 3. 前端接入：用深链唤起客户端

业务前端要做的只有一件事：**拼出一个 `swiftexport://` 链接并让浏览器打开它**。

### 3.1 链接格式

支持两种等价写法，二选一：

**写法 A（推荐）——直接给完整流地址：**

```
swiftexport://export?job=<jobId>&url=<完整流地址(必须 encodeURIComponent)>
```

**写法 B——给基址 + 令牌，由客户端拼接：**

```
swiftexport://export?job=<jobId>&host=<服务端基址>&token=<令牌>
```

> 写法 B 时客户端按固定模板拼接：`{host}/shy/export/stream?job={job}&token={token}`。
> 如果你的接口路径不是 `/shy/export/stream`，请用**写法 A** 自行给全 `url`。

### 3.2 参数说明

| 参数 | 必填 | 含义 | 注意 |
|------|------|------|------|
| `job` | 是 | 本次导出任务 id。用于：去重（同 job 进行中会被忽略）、多文件子目录命名、任务卡片展示。 | 建议用服务端生成的唯一任务号。 |
| `url` | 写法 A 必填 | **完整**的 Arrow 流地址，含鉴权参数。 | ⚠️ **必须 `encodeURIComponent` 整体编码**，否则内部的 `?`/`&` 会被当成深链自己的参数而截断。 |
| `host` | 写法 B 必填 | 服务端基址，如 `https://api.example.com`。 | 末尾 `/` 会被自动去掉。 |
| `token` | 写法 B 必填 | 鉴权令牌，拼进 query。 | 建议**一次性 / 短时效**，由服务端签发与校验。 |

> 🔒 **安全**：`url` 内含鉴权信息，客户端**仅内部使用、绝不在界面展示**；客户端产生的任何
> 错误提示也**不会回显 URL**（不泄露 job/token）。请把 token 设计成短时效一次性令牌。

### 3.3 在网页里触发

最简单——直接跳转当前页（不会真正导航离开，浏览器会拦截并唤起客户端）：

```js
// 写法 A：服务端已返回带鉴权的完整流地址 streamUrl
function openExport(jobId, streamUrl) {
  const link = `swiftexport://export?job=${encodeURIComponent(jobId)}`
             + `&url=${encodeURIComponent(streamUrl)}`;   // 整体编码，关键！
  window.location.href = link;
}

// 写法 B：只有基址 + 令牌
function openExportByToken(jobId, host, token) {
  const link = `swiftexport://export?job=${encodeURIComponent(jobId)}`
             + `&host=${encodeURIComponent(host)}`
             + `&token=${encodeURIComponent(token)}`;
  window.location.href = link;
}
```

典型用法：点击「导出」按钮 → 调你自己的后端开一个导出任务、拿到 `jobId` 和带鉴权的 `streamUrl`
→ 调 `openExport(jobId, streamUrl)`。

```js
async function onClickExport() {
  // 1) 让你的后端准备一次导出（鉴权、生成一次性 token、拼出流地址）
  const { jobId, streamUrl } = await fetch('/your-api/export/prepare', {
    method: 'POST', body: JSON.stringify({ /* 你的查询条件 */ }),
  }).then(r => r.json());

  // 2) 唤起客户端
  openExport(jobId, streamUrl);
}
```

### 3.4 几个你**不用**自己处理的边界（客户端已兜底）

- **客户端没开**：浏览器唤起会冷启动客户端，第一条深链会被客户端「收件箱」缓冲，待界面就绪后领取——**首次点击不会丢任务**。
- **客户端已在运行**：第二次唤起经单实例机制转发给已有窗口并聚焦——**不会开多个进程**。
- **用户狂点 / 同条链接重复投递**：同一 `job` 正在进行中会被自动忽略——**不会重复导出**。

所以前端**无需**做重试、无需轮询、无需判断客户端是否在运行。

### 3.5 客户端未安装的兜底

浏览器无法 100% 可靠地探测自定义协议是否注册。实践建议：

- 在导出入口附近放一个「未安装？下载客户端」的链接，指向仓库 Releases；
- 或唤起后用一个超时兜底提示（如 2~3 秒内页面无失焦，则提示去下载），但这只是体验优化，非强约束。

## 4. 后端接入：Arrow 流式导出接口（契约）

这是接入的**核心契约**。客户端对「导出内容」零硬编码——它把一切交给你在 Arrow schema 里描述。

### 4.1 接口形态

| 项 | 约定 |
|----|------|
| 方法 | `GET`（客户端用 `ureq::get(url)`，无请求体） |
| 路径 | 任意（写法 B 默认 `/shy/export/stream`，写法 A 可为任意 `url`） |
| 鉴权 | 自定（建议 query 上的一次性 `token`，服务端自行校验） |
| 成功响应体 | **Arrow IPC Stream** 格式的二进制字节流（`StreamReader` 可读）；支持 **LZ4 / ZSTD 压缩**。 |
| `Content-Type` | **不能含 `json`**（建议 `application/octet-stream` 或 `application/vnd.apache.arrow.stream`）。含 `json` 会被当成业务错误体。 |
| `X-Export-Total-Orders` | 可选响应头：顶层组（订单）**总数**估计，整数。用于进度条/ETA；缺省或 `0` → 客户端只显示吞吐、不显示 ETA。 |
| 传输 | 建议 **chunked 流式输出**：服务端边查边写边发，自身也不积压内存。 |

### 4.2 错误契约

业务错误请用 **JSON** 返回，客户端会提取其中的提示文案展示：

- HTTP `4xx/5xx` + JSON 体；**或**
- HTTP `200` + `Content-Type: application/json` + JSON 体（适配「全局异常处理器把 `ApplicationException` 包成 200 + Result」的后端）。

JSON 里任一字段会被取作错误信息（按序）：`message` → `error` → `msg`。

```json
{ "message": "导出数据为空，请调整筛选条件" }
```

> 传输层错误（DNS/连接/超时）客户端统一提示「网络连接失败」。**所有错误提示都不含 URL。**

### 4.3 自描述 Schema（决定 Excel 长什么样）

客户端从 **Arrow schema 的 metadata** 读取全部排版信息：

**① Schema 级 metadata**

| key | 含义 |
|-----|------|
| `sheet_title` | 顶部大标题（整行合并），如「订单列表」。可选，无则不画标题行。 |

**② 字段（Field）级 metadata —— 用 `role` 区分两类列**

**`role = "gid"`** —— 分组/层级 id 列（**类型必须 `Int64`**），不进 Excel，只用来判定分组边界：

| key | 含义 |
|-----|------|
| `role` | 固定 `"gid"` |
| `level` | 该层深度，**0 = 顶层（一条「订单」）**，向下递增。 |
| `group` | 该层的「组头」文案（如 `子订单`、`明细`），渲染为横跨该层列的表头。第 0 层留空。 |

**`role = "col"`** —— 数据列（**类型必须 `Utf8`/String**）：

| key | 含义 | 默认 |
|-----|------|------|
| `role` | 固定 `"col"` | — |
| `title` | 列表头文字 | 空 |
| `merge` | `"true"`/`"false"`：是否在本层组内**纵向合并**该列单元格 | `false` |
| `level` | 该列归属的层级（0=顶层） | `0` |
| `width` | 列宽（浮点） | `12.0` |

**③ 两条硬性数据约束**（违反会导致 panic 或合并错乱）：

1. **所有数据列必须是字符串**。金额、日期、数量等请在**服务端预格式化成展示文本**
   （如 `"¥1,234.50"`、`"2026-06-22"`），客户端不做任何类型转换。
2. **行必须按分组排好序**：同一组的行必须**物理连续**。客户端**不排序**，靠比较相邻行的
   `gid` 来判组边界——乱序会把一个组拆成多段、合并出错。SQL 侧请 `ORDER BY 顶层id, 次层id, …`。

### 4.4 二层示例（订单 → 明细）

假设要导出「每个订单含若干明细行」，订单级字段在明细行上重复、需要纵向合并：

Arrow schema（字段顺序即列顺序）：

| 字段名 | 类型 | role | level | title | merge | group |
|--------|------|------|-------|-------|-------|-------|
| `__gid0` | Int64 | gid | 0 | — | — | （空） |
| `order_no` | Utf8 | col | 0 | 订单号 | true | — |
| `customer` | Utf8 | col | 0 | 客户 | true | — |
| `amount` | Utf8 | col | 0 | 订单金额 | true | — |
| `__gid1` | Int64 | gid | 1 | — | — | 明细 |
| `sku` | Utf8 | col | 1 | 商品 | false | — |
| `qty` | Utf8 | col | 1 | 数量 | false | — |

schema metadata：`sheet_title = "订单列表"`。

渲染效果（订单级三列在该订单的所有明细行上合并；「明细」作为组头横跨 商品/数量）：

```
┌─────────────────────────────────────────────────────────┐
│                       订单列表                            │  ← sheet_title 整行合并
├──────────┬────────┬──────────┬──────────────────────────┤
│          │        │          │           明细           │  ← level1 组头 group=明细
│  订单号  │  客户  │ 订单金额 ├────────────┬─────────────┤
│          │        │          │    商品    │    数量     │  ← 各列表头
├──────────┼────────┼──────────┼────────────┼─────────────┤
│          │        │          │  T 恤      │     2       │
│ DD20260601│ 张三  │ ¥259.00 ├────────────┼─────────────┤  ← 订单三列在 2 行明细上合并
│          │        │          │  袜子      │     5       │
├──────────┼────────┼──────────┼────────────┼─────────────┤
│ DD20260602│ 李四  │  ¥99.00 │  水杯      │     1       │
└──────────┴────────┴──────────┴────────────┴─────────────┘
```

数据行（按 `__gid0, __gid1` 排序后）：每条明细一行，订单列重复值，客户端按 `__gid0` 变化判订单边界并合并。
> 三层及以上同理：再加 `__gid2`(level=2) + 对应 level=2 的列即可，合并逐层进行。

### 4.5 Java 服务端要点

用 `arrow-java` 写 IPC Stream（与客户端注释提到的 `ArrowExportWriter` 互通）：

1. 构造 `Schema`，给 schema 和每个 `Field` 挂上面表里的 **metadata**（`role/level/title/merge/width/group/sheet_title`）。
2. `gid` 列用 `BigIntVector`（Int64），数据列用 `VarCharVector`（Utf8，值已格式化为字符串）。
3. 用 `ArrowStreamWriter` 分批（`VectorSchemaRoot` 多个 batch）**边查边写边 flush** 到 HTTP 响应输出流。
4. 可开启 IPC 压缩（LZ4_FRAME / ZSTD）减小传输量——客户端已带 `ipc_compression` 支持。
5. 在响应头写 `X-Export-Total-Orders`（订单总数，可先 `COUNT` 估算）。
6. 数据 SQL 务必 `ORDER BY` 各层 id，保证同组连续。

## 5. 客户端行为（业务方需知道的）

| 行为 | 说明 |
|------|------|
| 输出文件名 | 由客户端命名为 `导出_<日期>.xlsx`（当前不可由调用方指定）。多文件时为 `导出_<日期>_partNN.xlsx`。 |
| 单文件 vs 多文件 | 单文件 → 直接落在下载目录根（重名自动加序号）；多文件 → 放在 `下载目录/<job前8位>/` 子目录。 |
| 分块粒度 | 「每文件订单数」`orders_per_file`，用户可在设置里调（默认 7 万）；越大文件越少但峰值内存越高（客户端给了内存预估提示）。 |
| 进度与 ETA | 客户端按 batch 实时更新「已导出 N 单 / M 行」，有 `X-Export-Total-Orders` 时显示剩余时间，否则显示「行/秒」吞吐。 |
| 任务历史 | 保留最近 10 条任务卡片，可点击「打开/打开目录」定位产物。 |
| 常驻 | 托盘常驻 + 可开机自启，便于随时被深链唤起。 |

## 6. 联调 Checklist

- [ ] 浏览器地址栏直接粘 `swiftexport://export?job=test&url=<编码后的流地址>` 回车，能唤起客户端并开始导出。
- [ ] 服务端可先把 Arrow 流**存成本地文件**，用 CLI（`xwjd-export-cli` 的 `generate_local`）先验证 schema/合并是否符合预期，再接到 HTTP 接口。
- [ ] 确认 `gid` 列是 Int64、数据列是 Utf8（**类型不符客户端会 panic**）。

- [ ] 大数据量压测：观察客户端内存是否随分块稳定（不随总量线性增长）。

## 7. FAQ

**Q：能指定导出的文件名吗？**
A：当前文件名固定为 `导出_<日期>`，由客户端控制。如需按业务自定义，需在客户端侧扩展（深链增加 `name` 参数）——可提 issue。

**Q：必须用 `/shy/export/stream` 这个路径吗？**
A：不必。那只是「写法 B」的默认拼接模板。用「写法 A」直接给完整 `url` 即可用任意路径。

**Q：token 怎么保证安全？**
A：建议服务端签发**短时效、一次性**令牌，校验后即失效；客户端不展示也不持久化 URL。

**Q：数字/日期为什么要服务端转字符串？**
A：客户端为「零业务硬编码」只认 Utf8 列、原样写入，不做本地化/格式化，避免精度与格式歧义。展示格式的主动权完全在服务端。
